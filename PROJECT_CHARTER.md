# 量化交易系统项目协作章程

版本：0.1  
日期：2026-07-04

## 1. 项目总目标

本项目的目标不是凭空重造一个庞大的量化私募级系统，而是在理智、客观、可验证、可复盘的前提下，逐步搭建一套属于自己的量化交易研究与模拟验证系统。

核心方向：

- 参考并学习 `daily_stock_analysis` 的成熟能力：数据聚合、LLM 诊股、报告生成、Web 工作台、通知推送。
- 优先复用成熟开源工具，而不是从零手写回测框架、撮合引擎、券商接口和底层交易基础设施。
- 用传统量化方法负责批量计算、因子筛选、回测验证和风控约束。
- 用 LLM 负责新闻、公告、财报、政策、舆情等非结构化信息的诊断、解释和风险排雷。
- 先跑通模拟盘闭环，再讨论真实资金执行。

重要原则：本项目不承诺、也不能保证最终盈利。我们能追求的是更严谨的研究流程、更低的主观误判、更强的复盘能力，以及长期验证后的概率优势。

## 2. 四人协作角色定位

### 用户

角色：CEO / 最终拍板人

职责：

- 决定项目方向、阶段目标和优先级。
- 决定风险承受范围、资金边界和是否进入下一阶段。
- 在 OpenClaw、Claude、Codex 观点冲突时做最终取舍。

### OpenClaw

角色：产品经理 + 金融知识顾问 + 风控挑战者

职责：

- 负责金融逻辑、投资常识、策略合理性和风控视角。
- 持续提醒团队避免盲从、过度工程化和需求膨胀。
- 对 Codex 和 Claude 的方案进行红队质疑。
- 推动 MVP 尽快验证，而不是陷入无休止造轮子。

### Claude

角色：项目计划设计者 / 路线规划师

职责：

- 拆分阶段、里程碑、验收标准和优先级。
- 设计实验方案、模拟盘验证流程和复盘口径。
- 帮助把宏观目标转化为可执行计划。

### Codex

角色：工程实践者 / 技术实现与验证

职责：

- 负责落代码、接工具、写适配层、跑测试和验证实现。
- 优先复用现有开源项目和成熟库。
- 守住工程边界：可测试、可审计、可复盘、默认不碰真实资金。
- 在实现中主动指出技术风险、数据风险和过度设计风险。

## 3. 核心工程纪律

1. 先验证，后扩张。
2. 先模拟，后实盘。
3. 先复用，后自研。
4. 先结构化信号，后自动化执行。
5. 先复盘统计，后资金放大。
6. 先承认不确定性，后讨论盈利可能。

## 4. 严格红线

- 不从零手写撮合引擎。
- 不从零手写完整回测框架。
- 不从零手写交易所或券商底层接口。
- 不让 LLM 直接控制真实资金。
- 不在缺少模拟盘结果、风控规则、审计日志之前进入实盘自动交易。
- 不用单次成功案例证明策略有效。
- 不因为回测结果漂亮就默认策略可实盘。
- 不把“高度拟合历史数据”当成目标，必须警惕过拟合。

## 5. 第一阶段 MVP 目标

第一阶段目标是快速验证金融逻辑和信号闭环，而不是搭建完整量化平台。

建议闭环：

```text
自选池 / 缩量标的池
-> 数据与新闻采集
-> LLM 诊断与风险排雷
-> 结构化交易信号
-> 模拟盘记录
-> 后续收益跟踪
-> 每日/每周复盘报告
```

第一阶段可以直接参考、Fork 或借用 `daily_stock_analysis`：

- 使用其数据源、报告、推送、Web 工作台等成熟能力。
- 魔改 Prompt、报告模板、风险纪律和信号输出格式。
- 让 LLM 输出标准化 JSON 信号。
- 信号只进入模拟盘或纸面交易系统。
- 每次信号必须记录当时价格、理由、置信度、止损、止盈、失效条件和数据来源。

## 6. 推荐信号结构

每个交易信号至少包含：

```json
{
  "symbol": "AAPL",
  "market": "US",
  "direction": "buy | sell | hold | watch",
  "confidence": 0.0,
  "entry_price": 0.0,
  "stop_loss": 0.0,
  "take_profit": 0.0,
  "max_position_pct": 0.0,
  "valid_until": "YYYY-MM-DD",
  "invalid_conditions": [],
  "reasoning_summary": "",
  "risk_notes": [],
  "data_sources": [],
  "created_at": "ISO-8601"
}
```

## 7. 复盘指标

每个阶段都必须能被数据打脸。至少跟踪：

- 信号数量
- 胜率
- 盈亏比
- 平均收益
- 最大回撤
- 持仓周期
- 止损触发率
- 止盈触发率
- 空仓/观望质量
- 误判原因分类
- LLM 风险提示是否有效
- 量化因子与 LLM 判断是否冲突

## 8. 量化层与 LLM 层分工

量化层负责：

- 全市场扫描
- 因子计算
- 多因子排序
- 回测验证
- 组合约束
- 风险指标
- 模拟盘记账

LLM 层负责：

- 新闻解读
- 公告和财报摘要
- 政策和宏观情景分析
- 舆情风险判断
- 个股诊断报告
- 对量化候选股做定性复核
- 输出可读、可审计的交易理由

推荐路线：

```text
量化层粗筛
-> 回测层验证
-> LLM 层精审与排雷
-> 风控层裁决
-> 模拟盘记录
-> 复盘评估
```

## 9. 开源工具使用原则

优先评估和复用：

- `daily_stock_analysis`：LLM 诊股、日报、推送、Web 工作台。
- AlphaSift：多因子选股与全市场扫描。
- AlphaEvo：策略验证与进化。
- Backtrader / vectorbt：回测与研究。
- vn.py：交易系统框架与网关生态。
- CCXT：加密货币交易接口。
- TradingView Webhook：轻量信号触发。

原则：我们写策略逻辑、适配层、信号标准、风控编排和复盘系统；成熟基础设施优先调用，不轻易重写。

## 10. 阶段推进原则

### Phase 1：拿来主义 MVP

- 快速跑通诊股、信号、模拟盘、推送和复盘。
- 不重写大框架。
- 不接真实资金。

### Phase 2：接入理科脑

- 接入成熟因子筛选或回测工具。
- 用缩量标的池降低复杂度。
- 验证量化粗筛 + LLM 精审是否优于单独使用任一方法。

### Phase 3：模拟盘强化

- 建立更完整的模拟盘、风控队列和审计日志。
- 对信号做长期统计。
- 研究不同市场环境下的稳定性。

### Phase 4：实盘前评审

- 只有在模拟盘表现稳定、风控规则明确、人工审批机制存在时，才讨论真实资金。
- 实盘执行必须单独审批，不默认自动开启。

## 11. 决策文化

- 直言不讳，拒绝盲从。
- 允许互相否定，但必须给出理由。
- 对市场保持敬畏。
- 对回测保持怀疑。
- 对 LLM 保持利用但不迷信。
- 对工程保持克制。
- 最终用长期记录和数据说话。
