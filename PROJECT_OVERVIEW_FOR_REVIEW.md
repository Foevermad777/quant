日期：2026-07-06　｜　作者：Claude（计划与验收角色）　｜　项目根：`~/quant`
用途：向 OpenClaw（产品/金融/风控红队）汇报当前进展、整体架构、路线预期，并列出请其挑战的开放问题。

---

## 一、项目是什么、不是什么

**是**：一套个人的量化交易「研究 + 模拟盘验证」系统。目标是把投资从主观判断，变成**可重复、可度量、可被数据打脸**的流程。
**不是**：一个承诺盈利的系统。我们追求的是更严谨的研究流程、更低的主观误判、以及长期验证后的概率优势——短期大概率不赚钱是基础概率，与选什么工具无关。

四方协作：CEO（拍板/风险边界）、OpenClaw（金融逻辑/风控红队）、Claude（计划/验收）、Codex（工程实现）。

---

## 二、核心架构决策：外挂封装，不深度 fork

参考开源项目 `daily_stock_analysis`（DSA，MIT）。经多智能体深度调研后定：**DSA 当后端服务用，我们只写差异化的薄层**。

- **DSA 负责**（一行源码不改，纯配置）：多源行情、中文新闻搜索、LLM 诊股、结构化信号库、模拟盘记账底表、复盘后验、多渠道推送。
- **我们自研**（核心资产，可整体搬迁）：① 沪深300 缩量池筛选器；② 注入「三大机械纪律」的 LLM 诊股层；③ 信号→模拟盘执行器；④ 复盘聚合。
- **为什么不 fork**：我们要改的东西（prompt、信号 schema、纪律）恰好落在 DSA 耦合最深处，且上游高频迭代，深 fork 必然与上游脱钩、合并成本失控。DSA 官方支持外挂调用（REST 信号写入 API），外挂是受支持的正道。

## 四、整体设计与终局（回答「量化系统真正的意义」）

量化系统的价值不是「选股」或「择时」任一单项，而是它们底下的引擎：**系统化 + 可度量 + 有纪律 + 可规模化**。选股(A)和择时(B)都只是这台引擎的输出。

**当前阶段本质**：先在 5 只固定股上，把「择时→执行→打分」这台**度量仪器**标定可信（= B）。**选股引擎(A)尚未建**——这是刻意的：没有可信的度量仪器，选股做得再花哨也无法判断真假、极易过拟合。这是章程「先验证后扩张」的纪律。

**终局是 A+B 合体**（章程第 8 节）：
```
量化层全市场扫描 + 因子排序 (A 选股/广度)
  → 回测验证 → LLM 精审排雷 (择时/质量)
  → 风控裁决 → 模拟盘记账 (B 执行/度量) → 复盘学习
```

诚实判断：**只在大盘股上择时，edge 很薄；更大的 alpha 可能在选股层(A)**。所以 A 不是不重要，而是被排在「度量骨架被证明可信」之后——先建能验证的地基，再上更有价值但更容易自欺的选股层。

## 六、自选池策略与规模上限

**当前**：5 只验证池——600519 茅台 / 300750 宁德 / 601318 平安 / 600036 招行 / 600900 长电（跨行业高辨识度大盘股）。

**积累期为何锁池**：14 天基线期间（现在→07-20）刻意维持这 5 只不动，理由是**样本一致性**——中途换池会让「这套系统到底有没有效」失去干净的对照，掉进「先验证后扩张」纪律要避免的坑。若有特别想盯的票，技术上加得进去，但会从加入日单独起算、把样本切碎，因此建议忍到复盘会一起扩。

**扩池的现实代价**（完整跑一轮含 LLM 诊股，当前串行 MAX_WORKERS=1）：

| 池子规模 | 单轮耗时 | LLM 成本（名义） | 主要瓶颈 |
|---|---|---|---|
| 5 只（当前） | ~7 分钟 | ~$0.07/天（免费额度内=0） | 无压力 |
| ~30 只 | ~40 分钟 | ~$0.4/天 | Bocha 新闻额度（每股~4次搜索，30只≈120次/轮） |
| 300 只（全沪深300） | ~6–7 小时 | 成本+限流爆炸 | 傍晚时间窗装不下 |

**规模上限**：无硬性代码上限，但有四道现实天花板——**时间**（串行跑，几十只已到傍晚窗口极限）、**配额**（Tushare 500次/天、Bocha 免费额度、免费爬虫反爬休眠累加）、**成本**（LLM 线性增长）、**人工质检**（CEO 每天能亲眼扫一遍报告判断「靠不靠谱」的量也就几十只）。**因此实际上限是「几十只」(~30-50)，不是几百只。**

**正确做法（章程既定）**：不把 300 只全喂 LLM（那正是章程明令禁止的 API 限流+上下文超载+成本爆炸），而是**量化层先把 300 筛成 30，再让 LLM 精审这 30**——这就是「缩量筛选器」里程碑存在的意义，也是选股层(A)落地的起点。所以「5 只 → ~30 只」是从「验证期配置」到「运行期配置」的自然演进，几百只是另一个需要分布式+付费数据源的量级，不在本架构路线上。

（注：标的池「选什么、选多少」本身是金融判断，与第九节红队问题 3 直接相关，欢迎 OpenClaw 从选股 edge 角度挑战。）

## 七、坚守的纪律与红线

- 先验证后扩张、先模拟后实盘、先复用后自研、先结构化信号后自动化执行、先复盘统计后资金放大。
- 不碰真实资金；不从零写回测/撮合/券商接口；不让 LLM 直控资金。
- 不用单次成功证明策略有效；对回测保持怀疑；警惕过拟合。
- 信号 schema、纪律、风控口径 = 我们的核心资产，永远自定义、可整体搬迁（换任何框架不受制于人）。
- **度量优先**：每条信号带理由/置信度/止损/失效条件，被 T+N 后验打分——系统每天都知道自己昨天准不准。

## 八、诚实的风险面与已知局限（请 OpenClaw 重点红队）

1. **三大机械纪律尚未注入**：数据溯源（精确发布日期/CY-FY）、术语规范化+失效条件、Base/Bull/Bear 多情景——目前跑的是 DSA 原生 prompt，属「纪律注入前的对照基线」。当前 LLM 已观察到的问题：把不相关新闻硬编成个股叙事、无媒体来源标注、旧闻当新利好。Phase 2 专治，但有效性未证。
2. **择时 edge 薄 + 选股层未建**：见第四节，当前系统的潜在价值上限受限。
3. **数据完整性**：免费源当日日线偶有缺口（已用 Tushare 大幅缓解 + 缺口补抓工具 + 晨检告警兜底）。
4. **配额天花板**：池子扩到几十只时 Bocha 新闻额度、LLM 成本、串行耗时（傍晚窗口）会成为约束；几百只不是本架构该走的路。
5. **置信度未校准**：当前 confidence 是「高/中/低」三档硬映射，非按真实胜率校准——Phase 2 加校准回路。
6. **单机 SQLite / 进程内状态**：并发弱，非高可用；日频够用，盘中级需另论。
7. **一切未经证实**：以上所有「设计上应该有效」的判断，都要等 14 天及后续复盘数据来证伪。

## 九、想请 OpenClaw 挑战的三个问题

1. **金融参数**：模拟盘默认「每信号固定 10% 等权、单股上限 20%、限价=信号 entry_high」——对 A 股大盘股的择时验证，这套仓位/成交假设是否合理？有无系统性偏差？
2. **复盘口径**：我们裁定「以护栏后的建议层为准、双基线用裸收益、小样本用 bootstrap 区间」。从金融统计角度，10 个交易日的样本能得出哪些「可信的」结论、哪些是噪音？第一次复盘会该看哪几个指标、警惕哪几个陷阱？
3. **选股层方向**：Phase 2 的缩量筛选器 + 因子排序，对沪深300 这类高效大盘股，缩量/多因子选股的 edge 现实吗？还是我们应该重新考虑标的池的选择（比如换更低效的中小盘）？
